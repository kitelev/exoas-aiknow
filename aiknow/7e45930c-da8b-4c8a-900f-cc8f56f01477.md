# Prototype = virtual aggregator

`ems__*Prototype` ассеты (TaskPrototype, MeetingPrototype, ProjectPrototype, EffortPrototype, SessionPrototype) — **виртуальные сущности для централизации и переиспользования связей**, не «шаблоны для копирования».

## Принцип

- **Prototype** хранит shared connections: `ems__Effort_area`, `exo__Asset_relates` (участники), `ems__Effort_isDefinedBy`, etc.
- **Instance** созданный из prototype через Create Instance кнопку, хранит **только**:
  - Свой `exo__Asset_uid`, `createdAt`, `label`, `aliases`
  - `exo__Instance_class` (целевой класс из Grounding_targetClass)
  - `exo__Asset_prototype` back-link на triggering prototype-instance (через executeCreateInstance)
  - `exo__Asset_isDefinedBy` (inherited unconditional)
  - `ems__Effort_status` = Draft (PropertyDefault)
- Запросы которым нужны shared properties — **следуют chain** `instance → exo__Asset_prototype → prototype → exo__Asset_relates / ems__Effort_area / etc.`

## Почему

Decompose-friendly. Пример: одна Area используется в двух MeetingPrototypes. User хочет разделить её на две Area. Достаточно изменить `ems__Effort_area` в двух prototypes — все instances этих prototypes автоматически отражают новое разделение. Если бы area копировалась в instances при создании, миграция требовала бы обновить N instances (а не 2 prototypes).

## Применение к Grounding design

Grounding `e01b025b` (Create MeetingPrototype instance, v 2026-05-25):

- ❌ НЕ копировать area, relates, прочие shared properties в instance frontmatter
- ✅ Только: isDefinedBy → isDefinedBy (через InheritanceRule `cbe000c4`) + executor пишет `exo__Asset_prototype` back-link автоматически
- ✅ `Grounding_targetFolder: "$targetFolder"` — instance лежит рядом с prototype (folder инхеритится from triggering asset's path, не из shared property)

## Anti-pattern

⛔ Копировать area/relates через Grounding_inheritanceRule (broken-by-design: рушит decompose-friendliness, instances «застывают» с устаревшими связями)

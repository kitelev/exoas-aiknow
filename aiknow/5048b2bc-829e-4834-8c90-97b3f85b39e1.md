# computer-control focus instability — activate before each keystroke

## Rule

При использовании `mcp__computer-control__computer` для UI-верификации, особенно длинных multi-step сценариев (Cmd+P → type → Enter → Cmd+P → ...), **никогда не доверять что target app остаётся в фокусе между tool calls**. macOS перебрасывает фокус между приложениями (наблюдал: Obsidian → Claude → Finder → Obsidian) даже без явных пользовательских действий.

**Defensive activate перед каждой keystroke-последовательностью:**

```bash
osascript -e 'tell application "System Events" to set frontmost of process "Obsidian" to true'
osascript -e 'tell application "Obsidian" to activate'
```

Затем минимум 1-2 sec sleep, затем keystroke. И даже это не гарантия — наблюдал, что queued keystrokes (от ранее заблокированной активации) появляются с задержкой в виде popup'ов quick switcher.

## Why

**Observed 2026-05-17** (v16.7.0 verification session):
- Послал Cmd+P в Obsidian → result.front_app="Claude" (фокус перехватил chat). Команда «Create fleeting note» не вызвалась.
- Послал Cmd+P через secs → front_app="Finder" (URI-открытие неявно дёрнуло Finder).
- Послал Cmd+Shift+B (ожидал backlinks) → активный таб переключился на posters/ExoPlugin EXO.
- В итоге ~6-8 keystroke последовательностей ушло в неправильные приложения за одну сессию верификации.

Cost: ~10 минут сессии и 4 рестарта computer-control flow.

## How to apply

1. **Перед длинной UI-верификацией** определи fallback'и заранее:
   - Можно ли проверить результат через file-content (`Read`) вместо UI?
   - Можно ли через Obsidian URI (`obsidian://open?vault=X&file=Y`) вместо нажатия клавиш?
   - Можно ли через DevTools console (`app.metadataCache.resolvedLinks`) вместо visual-inspection?

2. **Если UI всё-таки нужен:**
   - Перед КАЖДОЙ keystroke-последовательностью: `osascript activate` + `set frontmost`
   - Между активацией и keystroke: `sleep 1` минимум
   - Проверять `front_app` в результате каждого call и abort если несовпадает

3. **Trust your unit tests + functional equivalence** для функциональной проверки уже verified предыдущей версии — если v(N) изменил только source-of-truth (e.g. JSON-embedded → standalone property) и unit tests доказывают output-эквивалентность, computer-control regression test не обязателен.

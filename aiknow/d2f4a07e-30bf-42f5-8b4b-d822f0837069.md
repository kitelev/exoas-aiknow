# Rule

`npx <package-name>` (e.g. `@kitelev/exocortex-cli`) may trigger registry install of latest version, even when local node_modules has pinned binary. `npx <binary-name>` resolves node_modules/.bin first BUT may fall back to global PATH if not found.

**For pinned dependencies — call binary directly:**
```js
const CLI = path.join(REPO_ROOT, 'node_modules', '.bin', 'exocortex-cli');
execFileSync(CLI, ['sparql', 'query', ...]);
```

**Why:** verified empirically 2026-05-17 — `check-invariants.js` used `npx @kitelev/exocortex-cli sparql query ...`. node_modules pinned 15.101.0. Global installed 16.2.0. `npx <package-name>` invocation got v16 from global PATH (or registry-cached) → `sparql` subcommand dropped in v16 → script fails.

**How to apply:**
- CI scripts that depend on pinned CLI version: direct binary path, NOT npx
- Local dev iteration: same
- If you must use npx for flexibility: `npx --no-install <binary-name>` (force-resolve local only, fail if not found)
- Symptom: command works on one machine, fails on another; or worked yesterday breaks today after global npm install

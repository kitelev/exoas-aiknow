# Reference — UID-canon regression family

After RFC-004 UID-canon migration (commit `ca2d916` 2026-05-16 — `strip-aliases.py` Phase 3), `exo__Instance_class` and other class-reference fields в vault стали bare-UUID form `[[1b20a8f0-…]]` instead of alias-form `[[1b20a8f0-…|ems__Task]]`.

Multiple consumers expected symbolic form for string comparison → silently broke.

## Mitigation primitive (committed)

`WikiLinkHelpers.resolveSymbolic(value, resolver)` — extends existing `normalize` to also resolve bare-UUID wikilinks via DI'd resolver. Two callers' environments wire resolver differently:
- **Plugin** — `app.metadataCache.getFirstLinkpathDest(uid, "").frontmatter.exo__Asset_label`
- **CLI** — prebuilt `Map<string, string>` from `collectParsedFiles` over the vault

## ESLint rule decision

**Deferred.** Audit found ~15 LOW sites doing valid `.includes("ems__")` against programmatic / normalized input (status workflow normalizers, hardcoded class literals, SPARQL IRI bindings). A blanket rule would generate >10 `eslint-disable` site annotations — bigger scope than the fix itself. Per CLAUDE.md rule «defer if >20 false positives» — replaced with this audit doc + the centralized helper as deterrent. Future regressions get caught by sibling test `GenericAssetCreationService.isAreaClass edge cases › should resolve UID-canon` (TDD trigger if someone removes the resolver pathway).

# Bundle-diff is authoritative bisect signal

**Rule:** При диагностике runtime-регрессии в bundled artifact (Obsidian plugin `main.js`, любой esbuild/webpack output) — fetch released bundles обеих версий и сравнивать их (md5 + grep ключевых конструкций). Source diff `git log v_old..v_new` — вторичный сигнал.

**Why:** Release bundle ≠ source code. Возможные расхождения:
- Test-only коммит (новые fixture'ы, integration test) не меняет prod-бандл — `md5(v_old.main.js) == md5(v_new.main.js)`.
- ESBuild tree-shaking может выбросить dead code, изменения в нём не доходят до bundle.
- Side-effect free re-exports могут не попасть в bundle.
- В моём случае v15.155.0 и v15.155.2 main.js byte-identical (md5 совпали), хотя source diff показывал изменение в ShapeLoader.ts → bisect был сбит, я неправильно blamed ShapeLoader.

**How to apply:**
1. Build (или curl с GitHub releases) main.js обеих версий.
2. `md5 v_old.main.js v_new.main.js` — если совпали, source diff irrelevant, bisect к следующей версии.
3. `grep -oE 'process\.env\.[A-Z_]+|require\("[a-z/]+"\)|import\("[a-z/]+"\)' v_*.main.js | sort -u | diff` — точечно сужать на чём ломается на mobile.
4. Только когда найден bundle с расхождением — копать source diff.

**Empirical reference:**
- 2026-05-17, диагностика iOS plugin load failure: user bisect через BRAT указал v15.155.2 как первая broken — но md5(v15.155.0.main.js) == md5(v15.155.2.main.js). Реальный регрессионный коммит был между v15.155.2 и v15.156.0 (cfd31118, PR #3062, top-level `process.env.HOME`).
- Без bundle-diff проверки потратил 1-2 turn'а на ошибочную гипотезу (ShapeLoader dynamic imports).

**Cross-reference:** `verify-before-assert.md` — confident negative claims без mechanical verification ловятся пользователем. Bundle-diff = mechanical verification для runtime-регрессий.

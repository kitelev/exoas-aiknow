# Reference

`/Users/kitelev/vault-2025/assetspaces/<namespace>/` and `/Users/kitelev/vault-exodev/assetspaces/<namespace>/` are each clones of an **independent TBox git repo** named `kitelev/exocortex-<namespace>-ontology` (e.g. `exocortex-exocmd-ontology`, `exocortex-exo-ontology`).

`kitelev/exocortex-starter-kit` is a **separate** repo containing template content (same UUIDs but **disjoint commits**).

## Why this matters

Migrating files in starter-kit (e.g. PR #97 changed `exocmd/planning/22a6ba6b-…md`) does NOT auto-propagate to the TBox repo. Vaults pull from TBox, not starter-kit.

**Sync workflow:**
1. `cd /Users/kitelev/Developer/exocortex-development/exocortex-starter-kit && git pull origin main` (get latest)
2. `/bin/cp <source>/exocmd/<path>.md /Users/kitelev/vault-2025/assetspaces/exocmd/<path>.md`
3. `cd /Users/kitelev/vault-2025/assetspaces/exocmd && git add -A && git commit -m "..." && git push origin main`
4. `cd /Users/kitelev/vault-exodev/assetspaces/exocmd && git pull origin main` (sync second vault)

## Verified 2026-05-17

After PRs #3133/#3135/#3139 + starter-kit PRs #96/#97 — vault's exocmd assets remained on legacy `service_call` because the plugin v16.4.0 (which removed TS service registrations) was deployed while vault still had unmigrated groundings. End-to-end smoke caught this gap. Fix: manual propagation as above.

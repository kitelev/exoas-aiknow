## Rule
- При поиске ассетов класса X в Gate 0 / relationship search НЕ использовать `FILTER(CONTAINS(STR(?class), 'ims__Concept'))` — IRI хранится как UUID-based, substring-match всегда даёт 0 результатов
	- ❌ `?s exo:Instance_class ?class . FILTER(CONTAINS(STR(?class), 'ims__Concept'))` — broken
	- ✅ `?s exo:Instance_class <https://exocortex.my/ims__Concept-iri>` — direct IRI
	- ✅ Или join: `?s exo:Instance_class ?class . ?class exo:Asset_label "ims__Concept"` — by label

## Why
- Class wikilinks в frontmatter (`[[dda12c48-...|ims__Concept]]`) RDF-ифицируются как IRI с UUID, а не строка
- `STR(?class)` возвращает `https://.../dda12c48-6886-4624-8710-ed4ba92ce2b3`, не `ims__Concept`
- Substring match на 'ims__Concept' никогда не сработает → silent 0 results → false negative «концепта нет» → создание дубликата

## How to apply
- Перед SPARQL поиском ассетов класса — взять UID класса из `~/.claude/rules/exocortex-uids.md` (ims__Concept = `dda12c48-6886-4624-8710-ed4ba92ce2b3`)
- Или вообще опустить class filter и фильтровать только по label/aliases — все ims__Concept лежат в `03 Knowledge/concepts/` или `inbox/`, не критично
- Grep fallback: использовать `grep -rEi 'pattern1|pattern2'` (extended regex с `-E`), не `grep -ri 'p1\|p2'` — на BSD/macOS BRE alternation `\|` не работает

## Incident
- 2026-05-10: при поиске «постфикс/суффикс» концептов SPARQL вернул 0, grep тоже 0 (regex BSD-incompatible). Создал дубль `-oid (суффикс)` вместо link-to-existing `Postfix` (`f2a4eea5-5d74-4877-9cb6-af1d0bc24f0f`). Пользователь поймал.

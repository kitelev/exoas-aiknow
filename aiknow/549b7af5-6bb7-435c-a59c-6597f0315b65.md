# Rule

When auditing whether a service/class/function is "dead code" — `grep "new ServiceName"` (symbol-name search) is **necessary but not sufficient**. It misses:

- **Type-only imports** — files that import interfaces/types FROM the service module but never instantiate the class
- **Re-exports** — barrel files that pass through symbols
- **Dynamic imports / late-binding consumers**

## Workflow

1. **Grep instantiations:**
   ```bash
   grep -rn "new ServiceName\|ServiceName\." packages/ --include="*.ts" | grep -v test | grep -v dist
   ```
2. **Delete files + edit re-exports** в `index.ts`
3. **Run `tsc --noEmit --skipLibCheck`** — any orphaned `import type` from deleted module becomes hard error
4. Cleanup orphaned consumers (often UI components that import types only)

## Verified empirically 2026-05-17

Session deleted 4 dead services. Symbol-grep on `AnalyticsService` etc. showed 0 callers. But `tsc --noEmit` caught `packages/obsidian-plugin/src/presentation/views/BehavioralAnalyticsDashboard.tsx` — 251 LoC React component importing `AnalyticsPeriod`/`SleepAnalysis`/`TaskCompletionAnalysis`/`WeeklyPattern`/`BehavioralInsight` **types** (not service class itself), never rendered anywhere. Symbol-grep missed because import was for types.

## How to apply

- ALWAYS run `tsc --noEmit` immediately after deleting any module — treat as "discovery tool", not just "regression catcher"
- For larger cleanups (≥3 modules), run `npm run check:types` + `npm run lint` together — they catch different orphans
- Document orphaned consumers found via tsc in PR body — they're additional value, not embarrassments

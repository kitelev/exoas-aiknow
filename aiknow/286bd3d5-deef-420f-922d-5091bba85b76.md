# Verify state of CLI architecture перед vision/RFC claims

**Правило:** перед формированием architectural vision или RFC для `@kitelev/exocortex-cli` (или любого активно-эволюционирующего компонента Exocortex) — обязательно сделать audit-pass: grep recent RFC inbox + `git log` по relevant code paths за последние 6-8 недель. Не полагаться на snapshot прошлой сессии или априорные обобщения «как обычно устроен CLI».

**Why:** инцидент 2026-05-16, vision-document для exocortex-cli. Я уверенно утверждал «native TS-implemented commands + dynamic RDF-defined commands сосуществуют, apply их объединяет». Реальность (verified после pushback пользователя):

- RFC `94e520da-c6f7-48af-944c-51298d68da45` (2026-05-02, Status: Implemented audit 2026-05-09) — «Production-ready RDF-driven CLI command execution — killer feature exocortex». Цель: **все** команды через единый generic интерпретатор (precondition + grounding), TS-хардкод депрекейтится.
- Commits `87ca0b16`, `ad73197b` (последние 2 недели): schedule sunset of legacy `command start <path>` (T7.2 deprecation warning shipped, T7.3 removal scheduled v15.57.0+).
- В `command.ts` явный SUNSET-T7.3 комментарий: «pushing callers onto `dyncommand exec <uid>`».

Vision построенный на «two-categories» modeli — структурно неверен. Целевая архитектура — homoiconic, без TS/RDF split.

**Это — второй verify-before-assert violation в одной сессии** (первый: после merge PR #3112 сказал «релиз произойдёт автоматически» без verify Auto Release; зафиксирован в feedback `b68948f1-7ade-4d41-b339-9575d4c965fe`). Паттерн, не one-off — низкая граница confident assertion при high-velocity refactoring в проекте.

**How to apply:**

Перед vision / RFC / «как сейчас работает X» для Exocortex компонентов с активным dev — выполнить mechanical audit-pass:

```bash
# 1. Recent RFC inbox (последние 6-8 недель)
ls /Users/kitelev/vault-2025/03\ Knowledge/inbox/*.md | xargs ls -lt | head -30
grep -l "<component-keyword>" /Users/kitelev/vault-2025/03\ Knowledge/inbox/*.md | head -10

# 2. Git log по relevant paths
git log --oneline --since="2026-04-01" -- packages/<component>/src/ | head -20

# 3. Sunset / deprecation comments в исходниках
grep -rnE "SUNSET|deprecated|legacy|sunset" packages/<component>/src/ | head -15

# 4. Active RFC referenced в коде
grep -rE "RFC [0-9a-f-]{36}|RFC #?[0-9]+" packages/<component>/src/ | head -10
```

5-10 минут audit-pass = час сэкономленных rewrite + потерянного доверия пользователя.

Особо опасно: confident architectural framing («два класса команд», «X является primary способом»). Эти claims **формируют design space** последующего RFC — ошибка распространяется через весь документ. Должны быть verified **before** написания первой строки vision.

**Triggers:**
- «как сейчас устроено X» / «текущая архитектура Y» / «существующие категории Z в проекте Exocortex»
- Перед write-ом vision-документа или RFC asset
- Перед confident сравнением «было vs стало» для активно-эволюционирующего кода

**Anti-pattern:** опираться на знание из прошлых сессий о структуре CLI / plugin / vault без свежей проверки. Refactor месячной давности уже мог invalidate snapshot.

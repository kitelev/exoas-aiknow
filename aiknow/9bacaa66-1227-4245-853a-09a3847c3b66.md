# HARD STOP RULE для multi-phase child Claude prompts

**Rule:** Если prompt дочерней сессии содержит multi-phase delivery (e.g. «Phase 1a + 1b + T2.x + T3.x до v16.0.0» или «release через несколько PRs»), prompt обязан содержать explicit HARD STOP RULE block:

```
Ты НЕ ИМЕЕШЬ ПРАВА закрываться до тех пор пока ВСЕ N условий не выполнены ОДНОВРЕМЕННО:
1. <empirical check #1 через Bash>
2. <empirical check #2>
…

Перед каждым emit COMPLETED — обязательно перепроверь через Bash. Fail-any → exit 0 (НЕ закрывайся), продолжай работу.
Промежуточные completion checkpoints — emit PROGRESS, но НЕ COMPLETED. COMPLETED зарезервирован для финального success.
```

Плюс в конце prompt — verification скрипт (bash блок), который явно проверяет каждое условие через `[[ ... ]] && echo ✅ || { echo ❌; exit 0; }`. Без `exit 0` на fail-path child может попытаться quit anyway.

**Why:** 2026-05-16 первый CLI v16 child (`claude-cli-v16-impl-1778950770`) объявил COMPLETED после ОДНОГО PR (#3129 T1.7+T1.8, additive minor 15.178.0) — несмотря на план «Phase 1a+1b+T2.x+T3.x до v16.0.0». Child применил собственное scope-judgment, narrowed scope до самого простого slot, и закрылся. Был частичный успех (релиз 15.178.0, 4 TBox ассета), но цель v16.0.0 major не достигнута.

Второй child (`claude-cli-v16-continue-1778952292`) с HARD STOP RULE в prompt прошёл до v16.0.0 атомарно за 32 минуты (включая subagent работу 15m 43s). Без strict gate второй child бы тоже закрылся раньше.

**How to apply:**
1. **Trigger condition:** prompt содержит ≥2 sequential phase markers (Phase 1a/1b/Phase 2 etc.), OR множественные PRs планируются, OR specific version number ожидается published на npm/gh-release.
2. **HARD STOP block** — в верхней части prompt (после ⛔⛔⛔ visual marker), перечислить ВСЕ verify conditions явно. Не «when done» — конкретные shell-проверяемые предикаты.
3. **Verification скрипт** — перед `tmux send-keys ... /quit Enter` запустить bash блок с `[[ ... ]] && echo ✅ || { echo ❌ FAIL — continue work; exit 0; }`. `exit 0` на fail-path обязателен (если bash блок exits non-zero, claude может всё равно попытаться продолжить exit).
4. **BLOCKED escape hatch** — если работа объективно невозможна (external blocker — npm registry reject, etc.), child должен emit BLOCKED ping и остаться idle (просто end current response без exit). Parent решает retry/kill.
5. **Intermediate PROGRESS** — на каждом merged PR / phase complete emit PROGRESS, не COMPLETED. COMPLETED только когда все verify gates pass.

**Применимо к:** spawn-nested-claude prompts (особенно через `/spawn-nested-claude`), любые multi-PR releases, refactors spanning ≥2 commits. Не применимо к single-task spawns (closed asset, one PR research dump).

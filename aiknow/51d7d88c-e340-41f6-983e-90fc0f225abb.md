# Multi-grounding pattern в `exocmd__Command` сломан

`bb00efed` (Create Task Instance, universal) задумывался как один command с **N groundings** — по одному per prototype (TaskPrototype → ems__Task, ProjectPrototype → ems__Project, MeetingPrototype → ems__Meeting, FleetingNotePrototype → ztlk__FleetingNote, EventPrototype → exo__Event).

Тело command'а описывало pattern: «один Command + N Grounding (per `Grounding_targetPrototype`) + N Bindings (per `targetClass`)».

## Реальность

`packages/exocortex/src/services/CommandResolver.ts:899-924` `loadLinkedGrounding`:

```ts
const refTriples = await this.tripleStore.match(
  parentSubject,
  Namespace.EXOCMD.term("Command_grounding"),
  undefined,
);
if (refTriples.length === 0) return null;
const ref = refTriples[0].object;   // ← FIRST WINS (silent)
```

**Loader берёт ТОЛЬКО первый grounding** из всех `exocmd__Command_grounding` triples. Iteration order non-deterministic — зависит от triple-store insertion order. Никакой filtering by `Grounding_targetPrototype` match с triggering asset's class.

Last 4 groundings из 5 у `bb00efed` фактически dead code в runtime — никогда не выбираются.

## Empirical demonstration (2026-05-25)

`MeetingPrototype` asset `d179e666` (1-2-1 n.burkov) с binding 22093ca1 → bb00efed:
- Expected: `ems__Meeting` в `03 Knowledge/toos/` (через grounding `e01b025b` MeetingPrototype-variant)
- Actual: `ems__Task` в `03 Knowledge/inbox/` (через grounding `a6ef8fda` TaskPrototype-default — оказался первым в iteration order)
- File: `03 Knowledge/inbox/52a48948-417c-4637-baac-c220fe996cc2.md` имел `exo__Instance_class: [[1b20a8f0|ems__Task]]` вместо ожидаемого `ems__Meeting`.

## Workaround (текущий)

**Dedicated pattern**: 1 Command + 1 Grounding + 1 Binding **per prototype**. Single grounding на command — нет ambiguity на iteration order.

Example: `2f25fcac` «Create Meeting Instance» → один grounding `b87330e0` → ems__Meeting. Binding `9466c784` connects to `ems__MeetingPrototype`.

Аналогично: `654e2cfc` Create Project Instance (1 grounding b7a84c3c), `85c204fa` Create Session Instance, etc. Это work-by-design pattern в текущей кодовой базе.

## Forward-fix candidate

Modify `loadLinkedGrounding` чтобы:
1. Loaded ALL `Command_grounding` refs (не только [0])
2. For each grounding read `Grounding_targetPrototype`
3. Match against triggering asset's `exo__Asset_prototype` value OR class chain
4. Pick the grounding whose `targetPrototype` matches; fallback to first if none match

Это unlock universal multi-prototype commands (`bb00efed` semantics). До тех пор — использовать dedicated.

## Anti-pattern

⛔ Создавать command с >1 `exocmd__Command_grounding` пока loader не fix'нут. Все кроме первого dead code.
⛔ Полагаться на comment в body command (типа "Pattern: один Command + N Grounding") как на factual описание runtime — body описывал intent, не реализацию.

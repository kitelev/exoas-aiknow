# Spawned child Claude — PROGRESS pings unreliable, poll vault/PR state

## Правило

При orchestrating child Claude через `/spawn-nested-claude` — **не полагайся** на `child-ping-progress.sh PROGRESS / COMPLETED` pings как primary signal. Child часто их игнорирует, даже когда промпт явно инструктирует emit'ить на каждом milestone.

**Что reliable:**
- ✅ STARTED ping (через lifecycle hook installed в Step 4.5 spawn skill) — auto-emitted, не зависит от child compliance
- ✅ EXITED ping (через chained `child-ping-progress.sh EXITED` в Step 5) — fires on TUI graceful exit
- ✅ Audit log `~/.claude/audit/child-lifecycle.jsonl` — STARTED всегда есть

**Что НЕ reliable:**
- ❌ PROGRESS pings — child обычно не вызывает, даже с явной инструкцией
- ❌ COMPLETED ping — same, и часто завершается без exit (TUI остаётся в idle)

## Workaround — active polling parent-side

Parent должен periodically (каждые 3-10 мин) проверять external state instead of waiting for pings:

```bash
# Состояние tmux pane (что child сейчас делает / последний output)
tmux capture-pane -t <child-session>:0 -p -S -50 | tail -30

# Состояние PR (если child работает с github)
gh pr view <N> --repo <repo> --json state,mergeStateStatus,statusCheckRollup

# Состояние vault (если child создаёт ассеты)
ls -lt /Users/kitelev/vault-2025/03\ Knowledge/inbox/*.md | head -5

# Audit log
grep "<child-name>" ~/.claude/audit/child-lifecycle.jsonl | tail -5
```

## Why

- **Инцидент 2026-05-16 (fix #3115 child session):** Запустил child под ключ через `/spawn-nested-claude` с промптом инструктирующим emit'ить PROGRESS pings на 6 milestone events (worktree-ready, repro-confirmed, fix-landed, PR-opened, CI-green, merged). Получил только STARTED. Узнавал статус через tmux capture-pane + `gh pr view`. Child выполнил задачу correctly (PR merged за 13 мин), но без PROGRESS visibility — приходилось polling.
- Без active polling parent рискует "asleep at the wheel" — child может зависнуть, или закончить и parent узнает спустя 30+ мин.

## How to apply

- При spawn child под ключ → запланировать polling cadence в parent's mental model: проверять через 5 мин, потом 10, потом 15.
- Polling commands группировать в одно Bash invocation (parallel checks).
- Не embedding promise "I'll wait for COMPLETED ping" — это false reliability.
- Если задача длинная (>30 мин estimated) — можно использовать `/wake-me-in 15m` для periodic re-check.

# Post-merge fix verification — release timestamp vs mergedAt

## Правило

После merge PR с фиксом — НЕ предполагать, что `npm install <latest>` сразу получит фикс. Auto-release бежит на post-merge GitHub Actions cycle → новая версия публикуется через 2-5 мин ПОСЛЕ merge. **Старая `latest` версия, зарелизенная ДО merge, НЕ содержит фикс.**

**Always check:**
1. `gh pr view <N> --json mergedAt` → timestamp T₁
2. `gh release list --limit 5` → найти release с `createdAt > T₁`
3. Install **строго эту версию или новее**: `npm install -g <pkg>@<version>`
4. Verify через **empirical repro snippet** из issue body → exit 0

## Why

- **Инцидент 2026-05-16:** Issue #3115 SHACL prototype IRI. PR #3116 merged @ 10:34 UTC. Поставил v15.174.2 (`Latest` на момент install), но v15.174.2 был зарелизен @ 10:04 UTC — за 30 мин ДО merge. Repro продолжал падать → ложно заключил "fix не в CLI, только в plugin". Через grep CLI imports нашёл что shared module → значит fix есть, просто не в текущей версии. Проверил `gh release list` → v15.174.3 @ 10:38 UTC (после merge). Install 174.3 → exit=0.
- Cost: 1 ложный hypothesis, 1 wasted retest cycle, ~3 мин корректировки.

## How to apply

- Для любого «PR merged, обнови и проверь» workflow — **обязательно сверить release createdAt vs PR mergedAt**.
- Если auto-release pipeline ещё не закончился — sleep 2-5 мин и перепроверить `gh release list`.
- Empirical re-test (repro snippet exit=0) обязателен **до** declarations типа "fix landed, ready to use".
- В sessions с multi-package monorepo — также grep `import` chains чтобы убедиться что fix действительно достигает CLI (даже если фикс в plugin source).

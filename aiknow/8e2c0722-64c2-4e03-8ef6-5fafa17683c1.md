# Rule

When chaining `.replace(/\$tokenA/g, ...).replace(/\$tokenB/g, ...)` substitution tokens that share a common prefix (e.g. `$today` ⊂ `$todayStart`), order matters:

1. **More-specific (longer) first** — `$todayStart` before `$today`
2. **Word boundary `\b` on the specific one** — `/\$todayStart\b/g` not `/\$todayStart/g`

**Why:** verified empirically 2026-05-17 — CLI v16.2.0 substituted `$today` first → input `$todayStart` → after first replace became `2026-05-17Start` (literal "Start" remaining). v16.4.1 fixed by re-ordering and adding `\b`. Symptom: `ems__Effort_plannedStartTimestamp: 2026-05-17Start` instead of `2026-05-17T00:00:00`.

**How to apply:**
- Any DSL with `$varName` tokens — design tokens to be unambiguous OR enforce order in replacer
- Generalized: same applies to URL path patterns, regex alternation, etc.
- **Verify in installed bundle, not just source on main** — release pipeline lag can deploy stale version while source has fix
- Test case: add adversarial input `$todayStartX` and verify substitution result

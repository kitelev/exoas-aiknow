# Bundled feat!: PR для major breaking-change release

**Rule:** Для major release (X.0.0) с breaking changes — собирай single `feat!:` PR со всеми logically-related изменениями, а не множество sequential PR через несколько minor versions.

**Why:**
- 2026-05-16 CLI v16.0.0: advisor рекомендовал bundled approach. Реализовано через single PR #3130 (`feat!: CLI v16.0 Unix-style surface (RFC 8e83442b)`). Auto Release workflow детектировал `!` marker в commit message → автоматически сделал major bump 15.178.0 → 16.0.0. Атомарный merge + публикация за 7 минут.
- Альтернатива (9 sequential PRs — Phase 1a × 3 PR, Phase 1b × 4 PR, Phase 2 × 1 PR, Phase 3 × 1 PR) → churn в releases (15.179.0, 15.180.0, …, 15.187.0), плюс риск что `!` marker потеряется в одном из squash messages и release agent не сделает major bump. Также: intermediate version с частично removed legacy команд = broken state для anyone npm install в этот промежуток.
- Логически связанные breaking changes (removal legacy + introduction new surface) семантически — один атомарный change. Splitting violates principle of least surprise для downstream consumers.

**How to apply:**
1. **Trigger:** RFC планирует **major version bump** (X.0.0) + ≥2 logically related breaking changes (removals + additions сходящиеся в single release).
2. **Single worktree, single branch** (e.g. `feat/cli-v16-unix-surface`), single PR title с `feat!:` или `BREAKING CHANGE:` footer per Conventional Commits.
3. **Verify breaking marker** прежде merge: `git log feature-branch --grep '!' | head` — ожидается ≥1 match.
4. **Pre-merge:** проверь auto-release.yml workflow реагирует на breaking marker — `grep -E 'major|breaking' .github/workflows/auto-release.yml` (или эквивалент).
5. **Post-merge verify:** `gh run list --workflow "Auto Release" --limit 1 -R kitelev/exocortex` SUCCESS + `npm view <package> version` начинается с правильного major X.
6. **Не делай bundled** если: changes не logically related (e.g., refactoring + unrelated bug fix), ИЛИ если bundle > ~1000 LOC (CI risk, reviewer fatigue), ИЛИ если incremental delivery даёт реальную value пользователям между PR.

**Когда НЕ применять:**
- Minor / patch releases — продолжать sequential pattern (отдельные PR, отдельные releases).
- Major version с >10 unrelated changes — раздели на 2 major releases.

**Verified pattern на:**
- CLI v16.0.0: PR #3130, 33/33 CI green, 7m PR-merge-to-release wall-time, 0 broken intermediate versions.

**Anti-pattern (don't do this for major releases):**
- 9 sequential PRs к Phase 1a/1b/T2.x/T3.x каждый = 9 minor version bumps. Промежуточные версии (e.g. 15.180.0 с removed `batch` но без `find`) — broken для anyone install в этот промежуток. Plus: child Claude может narrow scope до single PR и закрыться (см. `[[9bacaa66-1227-4245-853a-09a3847c3b66]]`).
